# Impeccable — Design Context

This file captures **audience, brand, and constraints** for UI work in this repository. It is the source of truth for design skills; implementation details still live in `src/app/globals.css` and components.

## Design Context

### Users

Primary context is **internal tools and a personal workbench**: you or a small team using the app for day-to-day work. Sessions may be long; the UI should reduce noise and favor clarity over spectacle. Jobs to be done vary by feature, but the through-line is **efficient, focused work** without marketing fluff.

### Brand Personality

**Minimal, serious, understated** — closer to luxury and craft than to playful consumer apps. Confidence comes from spacing, type, and restraint, not from loud color or decorative chrome. Interfaces should feel **considered and durable**, like a well-made tool you reach for repeatedly.

### Aesthetic Direction

- **Visual tone**: spare hierarchy, generous whitespace used with intention, strong typographic contrast between levels, **accents used rarely** (aligned with existing “mostly neutral” philosophy in README/CLAUDE).
- **Theme**: **Light and dark are equally important** — ship both and treat the user’s theme choice as a first-class preference (not an afterthought).
- **Anti-references**: Avoid **generic “AI product” dashboards** — purple-on-dark clichés, interchangeable card grids, gradient text, decorative sparklines, and template-y admin patterns. Prefer distinctive structure and honest materials (solid type, real states, clear data hierarchy).
- **Existing palette note**: Current tokens in `globals.css` lean **deep navy, ocean teal, warm amber** on OKLCH neutrals; future tweaks should stay cohesive (tint neutrals toward the brand hue) rather than bolting on unrelated accent colors.

### Design Principles

1. **Restraint is the feature** — Add UI only when it improves comprehension or action; skip ornament that does not carry meaning.
2. **No template fingerprint** — If it could be any AI-generated SaaS demo, redesign until it could not.
3. **Parity across themes** — Components and pages must be legible and intentional in **both** light and dark modes.
4. **Density without clutter** — Internal tools can show data-rich surfaces; hierarchy and spacing carry the load, not extra boxes and stripes.
5. **Best-effort accessibility** — Sensible contrast, visible focus, and respect for `prefers-reduced-motion` without a formal WCAG certification requirement unless a feature demands it.

### Technical constraints (from repo)

- Next.js App Router, TypeScript, Tailwind CSS, shadcn/ui (customized, not stock).
- Live theme: `src/app/globals.css`; `src/styles/design-tokens.ts` is reference-only.
